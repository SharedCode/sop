# Knowledge Base (Spaces) Orchestration Manual
This manual defines the best practices and workflows for managing "Spaces" (Vector Databases and Long-Term Memory). You already have access to the exact API signatures and JSON schemas via your native tool definitions. This guide explains *how* and *when* to orchestrate them.

<h2> 1. The Space Architecture</h2>
A Space is a specialized Knowledge Base structured around **Categories** (topics/domains) and **Items** (individual context chunks or documents).
- **Categories** group related knowledge.
- **Items** contain the actual textual `content` and semantic embeddings.

<h2> 2. Orchestration Workflows</h2>

<h3> A. Research and Discovery (Read)</h3>
When you need to learn about a topic, do not guess or hallucinate. Use a progressive discovery pipeline:
1. **Explore Categories**: Use `list_space_categories` to see what high-level topics exist.
2. **Explore Items**: Use `list_space_items` to list what specific chunks exist within a category.
3. **Search Content**: Use `search_space` for semantic similarity queries to find the most relevant items across the space.

<h3> B. Data Mutation (Write)</h3>
When you are told to store knowledge, create documents, or save information:
1. **Generated Content Storage**: Use `mint_to_space` to write generated knowledge into a Space. Ensure `content` is highly cohesive and meaningful out of context.
2. **Required Target Space**: When the user names a Space, pass that exact Space name in `mint_to_space.kb_name` even if the Space does not exist yet.
3. **Generated Lists**: If the user asks for multiple generated entries, call `mint_to_space` once per logical item unless they explicitly want a single combined note.
4. **Import vs Generate Rule**: If the content is being generated by the model inside the current conversation, do NOT try to use an external import workflow. Generate the content, then store it with `mint_to_space`.
5. **Full Space Deletion**: Use `delete_space` to remove an entire Space when the user explicitly asks for that destructive action.

<h3> C. The Vectorization Lifecycle</h3>
Vectorizing computes the AI embeddings so data is searchable. **Do NOT call a vectorization tool unless the user explicitly asks for vectorization, embeddings, reindexing, or searchable semantic refresh.** Mutation alone is not sufficient reason to vectorize. When the user does explicitly request it, orchestrate the most precise tool for the job:
- Use `vectorize_space_items` to update specifically changed items.
- Use `vectorize_space_categories` to update an entire topic (if many items changed).
- Use `vectorize_space` only if triggering a full space reconciliation.
- Run vectorization as a single logical transaction in your reasoning steps where possible.

If the user only asks you to store, update, or delete knowledge, perform that mutation and stop there unless they also ask to vectorize.

<h3> D. Space Configuration (Routing & Persona)</h3>
Spaces can be configured with specific personas, routing rules, and tool access controls to specialize how the AI behaves within them.
1. **Reading Config**: Use `read_space_config` to see the currently enforced setup (System Prompt, Allowed Tools, Routing Hooks).
2. **Setting Config**: Use `update_space_config` to adjust the currently enforced setup (System Prompt, Allowed Tools, Routing Hooks).

<h2> 3. Transactionality Guardrails</h2>
Spaces are built on top of the broader database engine, meaning most data operations require explicit transactions.
- **Strict Tool Sequencing**: For transaction-managed space operations, open the database (`open_db`), then begin a transaction (`begin_tx`), execute the operation, and then commit the transaction (`commit_tx`).
- **Write mode**: Transaction-managed mutations like `delete_space_items` require `begin_tx(mode="write")`.
- **Direct Minting Exception**: `mint_to_space` manages its own transaction internally. Do NOT wrap `mint_to_space` in `open_db` / `begin_tx` / `commit_tx`.
- **Read mode**: Discovery operations (`list_space_categories`, `search_space`) require `begin_tx(mode="read")`.
- **Vectorization Exception**: The vectorization APIs (`vectorize_space_categories`, `vectorize_space_items`, etc.) manage their own internal transactional state. Do **NOT** invoke them inside an active transaction.
- **Delete Space Exception**: `delete_space` is a direct destructive operation. Prefer a confirmation step with the user before executing it, and do NOT wrap it in `begin_tx` / `commit_tx`.
